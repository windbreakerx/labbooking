# Labbooking — одностраничник для куратора и ИТ СПбГУ

**Назначение:** замена [lr.spmi.ru](https://lr.spmi.ru) — системы записи студентов на лабораторные работы.  
**Статус:** пилотный MVP, готовность ~75–90% (июнь 2026).  
**Контакт по продукту:** [указать ФИО, email разработчика / кафедры]

---

## 1. Зачем новый сервис, если портал уже есть

| | spmi.ru / studlab (Drupal) | lr.spmi.ru (сейчас) | labbooking (цель) |
|--|---------------------------|---------------------|-------------------|
| Задача | Сайт университета, информация | Операционная запись на ЛР | То же, но поддерживаемо |
| Пользователи | Все | Студенты, staff лабораторий | Те же |
| Стек | PHP 7.4, Drupal 9 (EOL) | PHP 8.2, кастомный код | Python 3.12, Django 5.1 |
| База | MySQL (типично для Drupal) | Своя БД | **Отдельная** PostgreSQL |

**Вывод:** labbooking не заменяет Drupal и не требует его обновления. Это **отдельный прикладной сервис** — по той же логике, что raspisanie, uspevaemost, priem на отдельных поддоменах.

Ранее обсуждался **отход от PHP** в пользу Go / Java / Python. Выбран **Python + Django** из‑за срока пилота (~3 месяца) и наличия разработчика с опытом в Python.

## 2. Что просим у ИТ (минимум для пилота)

| Компонент | Требование | Альтернатива, если не разрешат |
|-----------|------------|--------------------------------|
| Хост | Отдельная **Linux VM** (2 vCPU, 4 GB RAM) или согласованный хост | Временный пилот на внешней VM, DNS `pilot-lr.spmi.ru` → proxy |
| Python | 3.12 + venv или Docker **только на этой VM** | Bare metal: systemd + Gunicorn |
| База | **PostgreSQL 14+** (отдельная БД `labbooking`, localhost) | Обязательно; к MySQL Drupal не подключаемся |
| Redis | localhost:6379 | **Не обязателен:** кэш в PostgreSQL или rate limit в Apache |
| Веб | Apache **reverse proxy** на `127.0.0.1:8000` | SSL и домен — на стороне ИТ |
| Домен | `lr.spmi.ru` (после пилота) или `pilot-lr.spmi.ru` | — |
| SMTP | Внутренний relay вуза | Для уведомлений студентам |
| Cron | 2 задачи (час + понедельник) | См. раздел 5 |
| ПДн | Согласование размещения на серверах вуза | Акт / поручение кафедры |

**Не нужно:** доступ к БД Drupal, правки портала, общий MySQL, публикация Postgres/Redis в интернет.

## 3. Схема (как это встанет рядом с текущей инфраструктурой)

```
Интернет
   │
   ▼
Apache (как сейчас на *.spmi.ru)  ← SSL, виртуальные хосты
   │
   ├── spmi.ru, studlab…  →  PHP / Drupal  (без изменений)
   │
   └── lr.spmi.ru         →  proxy_pass → Gunicorn :8000 → Django
                                    │
                                    ├── PostgreSQL (только labbooking)
                                    └── Redis (опционально, localhost)
```

Студент в браузере видит привычный URL `https://lr.spmi.ru`. Меняется только **backend за прокси**.

## 4. Почему Python, а не «как у всех» PHP

| Аргумент | Пояснение |
|----------|-----------|
| Замена одного сервиса | Меняем lr.spmi.ru, не весь портал |
| Поддерживаемые версии | Django 5.1, Python 3.12 — с активной поддержкой безопасности |
| Срок пилота | Готовая auth, ORM, админка, миграции БД — быстрее, чем Go/Java с нуля |
| Соответствие ТЗ | Серверный HTML (HTMX), отчёты Excel, роли, изоляция по лабораториям — уже в MVP |
| Отдельная БД | PostgreSQL не конфликтует с MySQL Drupal |

Переписывание labbooking на PHP «для единообразия» **не сокращает** работу ИТ (всё равно отдельный хост, БД, прокси) и **откатывает** договорённость об отходе от legacy PHP на lr.

## 5. Эксплуатация (что должен знать дежурный)

**Процессы:** Gunicorn (3 worker), PostgreSQL, опционально Redis, фронт — Apache proxy.

**Cron (пример):**
```cron
0 * * * *   cd /opt/labbooking && venv/bin/python manage.py mark_visited
0 6 * * 1   cd /opt/labbooking && venv/bin/python manage.py generate_sessions --weeks=4
```

**Бэкап:** ежедневно `pg_dump` БД + каталог `media/` (загруженные PDF методичек).

**Обновление версии:** `git pull` → `migrate` → `collectstatic` → перезапуск Gunicorn (или `scripts/deploy-vm.sh` на VM с Docker).

**Healthcheck:** `GET /api/health/` — 200 OK.

**Логи:** Gunicorn stderr + Apache access/error для lr.spmi.ru.

## 6. Безопасность и ПДн (кратко)

- Обрабатываются: ФИО, email (`@stud.spmi.ru`), группа, записи на ЛР, обращения, PDF.
- Секреты только в `.env` на сервере, не в git.
- CSRF для web-форм, JWT для API, rate limit на логин и создание записи.
- Изоляция данных: студент видит только свой учебный план; staff — только свою лабораторию.
- SSO/LDAP — **после пилота** (`SSO_ENABLED=0` на старте).

## 7. План внедрения (без остановки портала)

| Этап | Срок | Действие |
|------|------|----------|
| 1 | 1–2 нед | Пилот на отдельной VM, реальные данные 1–2 лабораторий |
| 2 | 2–4 нед | Согласование с ИТ: VM, Postgres, поддомен, SMTP |
| 3 | 1–2 нед | Развёртывание на хосте вуза, параллельный прогон со старым lr |
| 4 | 1 день | Переключение DNS/vhost `lr.spmi.ru`, старый PHP — в резерв 2 нед |

Откат: вернуть Apache proxy на старый lr.

## 8. Ответы на типичные возражения ИТ

| Возражение | Ответ |
|------------|--------|
| «У нас только MySQL» | Нужен **отдельный** PostgreSQL на VM labbooking, не замена MySQL Drupal |
| «Не любим Docker» | Деплой возможен без Docker: venv + systemd + Gunicorn |
| «Зачем Redis» | Не обязателен для пилота; кэш можно в Postgres |
| «Сначала обновите Drupal» | Независимые проекты; lr не зависит от Drupal |
| «Слишком тяжело» | На пилоте ~сотни пользователей; 1 VM 4 GB достаточно (см. приложение в README) |
| «152-ФЗ» | Размещение на серверах вуза + согласованный SMTP + бэкапы |
