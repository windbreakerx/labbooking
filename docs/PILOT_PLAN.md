# Состояние labbooking и план пилота

Анализ по репозиторию (Django + DRF + HTMX) и ТЗ для студентов и сотрудников лаборатории ([students_tz.txt](students_tz.txt), [zavlab_tz.txt](zavlab_tz.txt)).

**Оценка готовности (ориентир на июнь 2026):**

| Контур | Готовность |
|--------|------------|
| Студенческий путь | ~85% |
| Панель сотрудника | ~70% |
| Изоляция данных по группе/лаборатории | ~35% |
| Соответствие ТЗ пилота | ~65% |
| До пилота при узком скоупе | 2–3 недели |

**Главный вывод:** фронт для студента и staff UI уже в приемлемом MVP-состоянии. Главный риск пилота сместился с базовой верстки на корректную видимость данных: студент должен видеть дисциплины своего учебного плана, сотрудники и завлабы — только свою лабораторию, дисциплины, ЛР, стенды, расписание и обращения. Для полной приёмки всё ещё не хватает интеграции с Деканатом/SSO и точных Excel-выгрузок.

См. также: [ROADMAP.md](ROADMAP.md) — общая дорожная карта по этапам.

## Что уже реализовано

| Контур | Реализовано | Оценка |
|--------|-------------|--------|
| Студент | Вход, список дисциплин активного семестра, список ЛР, календарная запись (дата → время → УЦ → аудитория), мои записи, отмена, обращения | Работает как MVP web UI и API; нужно ограничить дисциплины учебным планом группы |
| Бизнес-правила | 14 дней вперёд, открытие дня в 22:00, запрет отмены менее чем за 24 часа, 1 активная запись на дисциплину, вместимость, праздники | Ядро в `BookingService` и `session_availability` |
| Статусы | Записан, Неявка, Отменил запись, Повторный доступ, Посетил; история статусов; авто VISITED для завершённых слотов | Нужно проверить сценарии на реальных данных и тексты писем |
| Сотрудник | Список записавшихся, фильтры, смена статуса, ручная запись в web UI, стенды, ЛР с PDF, дисциплины, расписание, обращения, Excel-отчёты | UI приемлем для пилота; ручную запись нужно перевести с email/select на поиск студента и календарь |
| Интеграции и эксплуатация | Docker, PostgreSQL, Redis, Nginx, healthcheck, Swagger, CI, скрипты деплоя, CSV-импорт Деканата | SSO и API Деканата — заглушки; SMTP надо настроить для пилота |

**Стек:** Django 5, DRF, SimpleJWT, PostgreSQL, Redis (cache), Gunicorn, Nginx, HTMX, openpyxl.

**Архитектура:** `views (web + api) → BookingService → models → PostgreSQL`.

## Разрывы с ТЗ

| Приоритет | Область | Что отсутствует или неполно | Почему важно |
|-----------|---------|----------------------------|--------------|
| Критично | Учебные планы групп | Сейчас `UserProfile.group_name` — строка, а студент видит все опубликованные дисциплины активного семестра | Студенты будут видеть лишние дисциплины и ЛР |
| Критично | Лабораторная видимость | Принадлежность дисциплин/ЛР лаборатории не задана явно; staff scoping применён не везде, особенно в API | Сотрудники и завлабы могут увидеть чужие данные |
| Критично | Роль завлаба | Есть `LAB_ADMIN`, но нет отдельной роли завлаба с управлением только своей лабораторией и привязками | Завлаб должен сам вести сотрудников, преподавателей, дисциплины, ЛР, стенды и расписание |
| Критично | Ручная запись сотрудником | Web-форма есть, но студент ищется только по email, слот выбирается длинным списком | Сотрудники не знают почты студентов; нужен поиск по ФИО/группе и календарный выбор |
| Критично | Email/SMTP | Шаблоны есть, backend по умолчанию console; нет подтверждения вузового SMTP и доставки | Студенты не получат обязательные уведомления |
| Высокий | Данные пилота | Нужна подтверждённая CSV-матрица: группы, дисциплины, ЛР, лаборатории, сотрудники, преподаватели, расписание | Без этого нельзя проверить изоляцию данных на реальном наборе |
| Высокий | Расписание по ТЗ | Модель нечёт/чёт и генератор слотов есть, UI пока больше read-only/admin fallback | Главная недостроенная часть панели завлаба |
| Высокий | Отчёты | 3 Excel-файла есть, но состав не совпадает с ТЗ: нет расширенной истории статусов, пола, группы, должности, признака ручной регистрации с ФИО | Для отчётности пилота хватит частично, для приёмки — нет |
| Средний | Сортировка и фильтры staff | В ТЗ быстрый поиск/сортировка в каждом разделе; в web UI частично или отсутствует | Влияет на удобство, не блокирует узкий пилот |
| Средний | SSO и Деканат API | Заглушки и CSV; нет реального SSO/LDAP/OIDC и автосинхронизации | Можно отложить при пилоте на локальных аккаунтах и CSV |

## Критично для пилота

- Рабочий деплой (HTTPS или согласованный IP), бэкапы PostgreSQL, порядок обновления.
- Реальные данные: студенты, группы, учебные планы, дисциплины, ЛР, аудитории, сотрудники, преподаватели, лабораторные привязки и расписание минимум на 2 недели.
- Сценарий студента: войти → дисциплина → ЛР → записаться → мои записи → отмена за 24 ч → обращение.
- Сценарий сотрудника: записи своей лаборатории → фильтр → статус → ручная запись через поиск студента и календарь → ответ на обращение → Excel.
- Сценарий завлаба: добавить/привязать преподавателя или сотрудника → привязать дисциплину/ЛР/стенд/расписание к своей лаборатории → проверить видимость.
- Email на события: запись, отмена, неявка, повторный доступ, посещение.
- Доступ: студент видит только свой учебный план; сотрудник/преподаватель/завлаб не видит чужие лаборатории.

### Можно отложить после пилота

Полноценный SSO/OIDC, API Деканата, React SPA, идеальное соответствие всех staff-экранов скриншотам ТЗ, очередь ожидания как отдельный сценарий.

### Нельзя откладывать

Реальные данные, запись и отмена, статусы, ручные действия сотрудника, email, базовые выгрузки, эксплуатационная проверка стенда.

## Обновлённый план на 2–3 недели

| Срок | Фокус | Задачи | Приоритет |
|------|-------|--------|-----------|
| Дни 1–3 | Модель доступа | `StudentGroup`, матрица группа → дисциплины/ЛР, явные связи лаборатория → дисциплины/ЛР, роль завлаба | Критично |
| Дни 3–6 | Видимость student/staff/API | User-aware queryset для дисциплин, ЛР, слотов и обращений; единый staff/lab-head scoping; запрет прямого доступа к чужим URL/API | Критично |
| Дни 5–8 | Ручная запись | Поиск студента по ФИО/email/группе; календарный выбор слота как у студента; сохранение staff-обходов только для разрешённых правил | Критично |
| Дни 7–10 | Кабинет завлаба | Минимальные web-экраны для добавления сотрудников/преподавателей и привязок к лаборатории, дисциплинам, ЛР, стендам и расписанию | Критично |
| Дни 9–12 | CSV и данные пилота | Расширить seed/import: группы, учебные планы, staff/teacher bindings, лабораторные привязки, расписание; загрузить реальные CSV | Высокий |
| Дни 10–13 | SMTP, отчёты, эксплуатация | SMTP вуза, проверка писем, Excel-поля для пилота, VM/HTTPS, бэкапы PostgreSQL, smoke-test | Высокий |
| Дни 12–15 | Приёмка и hardening | Тесты видимости для студента/staff/завлаба/API, инструкции, багфиксы по результатам живого сценария | Критично |
| Неделя 3 | Буфер пилота | Полировка staff-поиска/сортировки, уточнение отчётов, подготовка к Деканату/SSO без включения их в обязательный скоуп | Средний |
