# Состояние labbooking и план пилота

Анализ по репозиторию (Django + DRF + HTMX) и ТЗ для студентов и сотрудников лаборатории ([students_tz.txt](students_tz.txt), [zavlab_tz.txt](zavlab_tz.txt)).

**Оценка готовности (ориентир на июнь 2026):**

| Контур | Готовность |
|--------|------------|
| Студенческий путь | ~90% |
| Панель сотрудника | ~80% |
| Изоляция данных по группе/лаборатории | ~80% |
| Соответствие ТЗ пилота | ~75% |
| До пилота при узком скоупе | 1–2 недели стабилизации |

**Главный вывод:** фронт для студента и staff UI уже в приемлемом MVP-состоянии, а базовая изоляция данных по учебной группе и лаборатории реализована в коде. Главный риск пилота сместился на проверку покрытия этой изоляции во всех web/API-точках, качество реальных CSV-данных, SMTP-доставку, эксплуатацию VM и удобство staff-сценариев. Для полной приёмки всё ещё не хватает интеграции с Деканатом/SSO и точных Excel-выгрузок.

См. также: [ROADMAP.md](ROADMAP.md) — общая дорожная карта по этапам.

## Что уже реализовано

| Контур | Реализовано | Оценка |
|--------|-------------|--------|
| Студент | Вход, список дисциплин активного семестра, список ЛР, календарная запись (дата → время → УЦ → аудитория), мои записи, отмена, обращения | Работает как MVP web UI и API; дисциплины/ЛР ограничены учебным планом `StudentGroup` |
| Бизнес-правила | 14 дней вперёд, открытие дня в 22:00, запрет отмены менее чем за 24 часа, 1 активная запись на дисциплину, вместимость, праздники | Ядро в `BookingService` и `session_availability` |
| Статусы | Записан, Неявка, Отменил запись, Повторный доступ, Посетил; история статусов; авто VISITED для завершённых слотов | Нужно проверить сценарии на реальных данных и тексты писем |
| Сотрудник | Список записавшихся, фильтры, смена статуса, ручная запись через поиск студента и календарь, стенды, ЛР с PDF, дисциплины, расписание, обращения, Excel-отчёты | UI приемлем для пилота; нужно проверить scoping всех staff-страниц и полировать UX |
| Завлаб | Роль `LAB_HEAD`, кабинет завлаба, сотрудники/преподаватели, привязки дисциплин/ЛР, стенды, расписание | Базовый web UI реализован; нужно проверить живой сценарий и границы доступа |
| Интеграции и эксплуатация | Docker, PostgreSQL, Redis, Nginx, healthcheck, Swagger, CI, скрипты деплоя, CSV-импорт Деканата | SSO и API Деканата — заглушки; SMTP надо настроить для пилота |

**Стек:** Django 5, DRF, SimpleJWT, PostgreSQL, Redis (cache), Gunicorn, Nginx, HTMX, openpyxl.

**Архитектура:** `views (web + api) → BookingService → models → PostgreSQL`.

## Разрывы с ТЗ

| Приоритет | Область | Что отсутствует или неполно | Почему важно |
|-----------|---------|----------------------------|--------------|
| Критично | Проверка scoping-покрытия | `StudentGroup`, лабораторные M2M-связи и `LAB_HEAD` реализованы, но нужно закрыть точечные разрывы во всех staff/web/API списках | Один незаскоупленный список может раскрыть чужие данные |
| Критично | Семантика `profile.disciplines` | Завлаб назначает дисциплины сотруднику/преподавателю, но текущие querysets ограничивают доступ по лаборатории, а не по персональному списку дисциплин | Нужно явно зафиксировать: это metadata для пилота или отдельная access-фича после пилота |
| Критично | Реальные учебные планы групп | Модель `StudentGroup` есть, но пилот зависит от корректной матрицы группа → дисциплины/ЛР и заполненного `student_group` у студентов | Ошибки CSV дадут пустую или неверную видимость |
| Критично | Ручная запись сотрудником | Поиск студента и календарный выбор уже реализованы | Нужно проверить UX на живых данных и сохранить staff-обходы только там, где это разрешено |
| Критично | Email/SMTP | Шаблоны есть, backend по умолчанию console; нет подтверждения вузового SMTP и доставки | Студенты не получат обязательные уведомления |
| Высокий | Данные пилота | Нужна подтверждённая CSV-матрица: группы, дисциплины, ЛР, лаборатории, сотрудники, преподаватели, расписание | Без этого нельзя проверить изоляцию данных на реальном наборе |
| Высокий | Расписание по ТЗ | Модель нечёт/чёт, генератор слотов и формы завлаба есть | Нужно проверить генерацию слотов и удобство редактирования на реальном расписании |
| Высокий | Отчёты | 3 Excel-файла есть, но состав не совпадает с ТЗ: нет расширенной истории статусов, пола, группы, должности, признака ручной регистрации с ФИО | Для отчётности пилота хватит частично, для приёмки — нет |
| Средний | Сортировка и фильтры staff | В ТЗ быстрый поиск/сортировка в каждом разделе; в web UI частично или отсутствует | Влияет на удобство, не блокирует узкий пилот |
| Средний | SSO и Деканат API | Заглушки и CSV; нет реального SSO/LDAP/OIDC и автосинхронизации | Можно отложить при пилоте на локальных аккаунтах и CSV |

## Критично для пилота

- Рабочий деплой (HTTPS или согласованный IP), бэкапы PostgreSQL, порядок обновления.
- Реальные данные: студенты, группы, учебные планы, дисциплины, ЛР, аудитории, сотрудники, преподаватели, лабораторные привязки и расписание минимум на 2 недели.
- Сценарий студента: войти → дисциплина → ЛР → записаться → мои записи → отмена за 24 ч → обращение.
- Сценарий сотрудника: записи своей лаборатории → фильтр → статус → ручная запись через поиск студента и календарь → ответ на обращение → Excel.
- Сценарий завлаба: добавить/привязать преподавателя или сотрудника → привязать дисциплину/ЛР/стенд/расписание к своей лаборатории → проверить видимость.
- Email на события: запись, отмена, неявка, повторный доступ, посещение.
- Доступ: студент видит только свой учебный план; сотрудник/преподаватель/завлаб не видит чужие лаборатории.

### Можно отложить после пилота

Полноценный SSO/OIDC, API Деканата, React/Next/Vue SPA, интерактивная карта аудиторий, идеальное соответствие всех staff-экранов скриншотам ТЗ, очередь ожидания как отдельный сценарий.

### Нельзя откладывать

Реальные данные, запись и отмена, статусы, ручные действия сотрудника, email, базовые выгрузки, эксплуатационная проверка стенда.

## Обновлённый план стабилизации на 1–2 недели

| Срок | Фокус | Задачи | Приоритет |
|------|-------|--------|-----------|
| Дни 1–2 | Scoping audit | Проверить student/staff/lab-head web/API списки, закрыть точечные разрывы, особенно staff people и прямые URL/API | Критично |
| Дни 2–3 | Реальные данные | Прогнать `seed_demo`, затем CSV-цепочку: группы, дисциплины, curriculum, lab bindings, staff/teacher bindings, students | Критично |
| Дни 3–4 | Ручная запись и staff UX | Проверить поиск по ФИО/email/группе, календарь ручной записи, статусы, обращения и базовые отчёты | Критично |
| Дни 4–5 | Кабинет завлаба | Проверить добавление людей, привязки, ЛР, стенды, расписание и отсутствие доступа к чужим лабораториям | Критично |
| Дни 5–7 | SMTP и эксплуатация | SMTP вуза, проверка писем, VM/HTTPS, бэкап PostgreSQL, cron, smoke-test | Критично |
| Дни 7–10 | Приёмка и hardening | Полный pytest, ручные сценарии student/staff/завлаб, исправления найденных P0/P1 | Критично |
| После пилота | Улучшения | Карта аудиторий, расширенные отчёты, SSO/API Деканата, возможный SPA | Средний |
