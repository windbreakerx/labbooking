# Дорожная карта labbooking

> Базовый MVP, scoping по группе/лаборатории, `LAB_HEAD`, HTMX staff UI, ручная запись и CSV-импорт реализованы. Ближайший фокус — стабилизация пилота: реальные данные, SMTP, VM/HTTPS, backup, staff UX и проверка границ доступа.

Опорный проект: **labbooking** (Django + DRF + PostgreSQL + Docker).  
Цель ближайшего этапа: **безопасный пилот на VM в Docker**, затем доработка по обратной связи и ТЗ СПГУ.

Подробный gap-анализ и план стабилизации пилота: [PILOT_PLAN.md](PILOT_PLAN.md).

| Где | Что делаете |

## Рекомендации по моделям AI (Cursor)

| Тип задачи | Модель | Режим |
|------------|--------|-------|
| Деплой, Docker, `.env`, скрипты | **Composer Fast** | Agent |
| План этапа, gap-анализ ТЗ | **GPT-5.5 Medium** / **Opus 4.8** | Plan / Ask |
| Django: модели, `BookingService`, API | **GPT-5.3 Codex** | Agent |
| Расписание, Деканат, SSO (архитектура) | **Opus 4.8** → затем Codex | Plan → Agent |
| Мелкие правки, тесты, lint | **Composer Fast** | Agent |
| Security перед prod | **Opus 4.8** + `.cursor/rules/security.md` | Ask |

**Правило:** один чат = одна фича; в начале чата: `@docs/ROADMAP.md` `@docs/zavlab_tz.txt`.

**Frontend:** до пилота не переносить HTMX на React/Vue/Next.js. Основной риск сейчас в данных, scoping, email, deploy и staff UX; SPA планировать отдельной вертикалью после стабилизации DRF-контрактов.

**Карта аудиторий:** не блокер пилота. Первая версия — простая read-only карта УЦ/этажа на HTMX/SVG/PNG с подсветкой аудитории; интерактивный редактор и маршруты — после живой обратной связи.
