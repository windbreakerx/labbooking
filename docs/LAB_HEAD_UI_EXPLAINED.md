# Lab Head UI: разбор кода

Этот документ объясняет, как устроен текущий код кабинета заведующего лабораторией.

Разбираемые файлы:

- `backend/apps/bookings/views/lab_head.py`
- `backend/apps/bookings/services/lab_head.py`
- `backend/templates/bookings/lab_head/*.html`
- `backend/apps/bookings/tests/test_lab_head_ui.py`

## 1. Что это за функциональность

Lab Head UI - это кабинет заведующего лабораторией.

Он нужен, чтобы пользователь с ролью `LAB_HEAD` мог управлять данными своей лаборатории:

- сотрудниками и преподавателями;
- дисциплинами;
- лабораторными работами;
- стендами;
- расписанием.

Основная идея:

```text
Завлаб входит в систему
  -> открывает /lab-head/...
  -> Django проверяет его роль и профиль
  -> код выбирает только данные его лаборатории или учебного центра
  -> шаблон рисует страницу
```

То есть завлаб не должен видеть все данные системы. Он должен видеть только то, что связано с его лабораторией или учебным центром.

## 2. Общая схема потока

Типичный путь запроса:

```text
Браузер
  -> URL /lab-head/lab-works/
  -> backend/config/urls.py
  -> backend/apps/bookings/urls_web.py
  -> backend/apps/bookings/views/lab_head.py
  -> backend/apps/bookings/services/lab_head.py
  -> база данных
  -> backend/templates/bookings/lab_head/lab_works.html
  -> HTML-страница пользователю
```

Простыми словами:

1. Пользователь открывает страницу.
2. Django находит подходящий URL.
3. URL вызывает нужную view.
4. View проверяет права.
5. View берет данные через service-слой.
6. Service-слой ограничивает данные лабораторией/УЦ завлаба.
7. View передает данные в шаблон.
8. Шаблон рисует HTML.

## 3. Файлы и их роли

### `backend/apps/bookings/views/lab_head.py`

Это файл с Django views.

View - это код, который отвечает на HTTP-запрос.

Например:

- открыть страницу ЛР;
- создать ЛР;
- обновить ЛР;
- привязать дисциплину;
- отвязать дисциплину;
- создать стенд;
- создать запись расписания.

Пример:

```python
class LabHeadLabWorksView(LabHeadRequiredMixin, ListView):
    template_name = "bookings/lab_head/lab_works.html"
    context_object_name = "lab_works"
```

Это значит:

- используется шаблон `bookings/lab_head/lab_works.html`;
- список объектов в шаблоне будет называться `lab_works`;
- доступ проходит через `LabHeadRequiredMixin`.

### `backend/apps/bookings/services/lab_head.py`

Это service-слой.

Он отвечает не за HTML, а за правила:

- кто считается завлабом;
- какую лабораторию видит пользователь;
- какой учебный центр видит пользователь;
- какие дисциплины доступны;
- какие ЛР доступны;
- как искать ЛР;
- как искать стенды;
- как обновлять ЛР;
- как проверить, что объект находится в зоне доступа.

Пример:

```python
def is_lab_head_user(user: User) -> bool:
    return user.is_authenticated and user.role == UserRole.LAB_HEAD
```

Простыми словами: пользователь должен быть залогинен и иметь роль `LAB_HEAD`.

## 4. URL-ы кабинета завлаба

Главный проектный URL-конфиг подключает `apps.bookings.urls_web`.

Дальше в `urls_web.py` есть маршруты:

```python
path("lab-head/", LabHeadHomeView.as_view(), name="lab-head-home")
path("lab-head/people/", LabHeadPeopleView.as_view(), name="lab-head-people")
path("lab-head/bindings/", LabHeadBindingsView.as_view(), name="lab-head-bindings")
path("lab-head/lab-works/", LabHeadLabWorksView.as_view(), name="lab-head-lab-works")
path("lab-head/stands/", LabHeadStandsView.as_view(), name="lab-head-stands")
path("lab-head/schedule/", LabHeadScheduleView.as_view(), name="lab-head-schedule")
```

Каждый URL ведет в свой класс из `views/lab_head.py`.

## 5. Пример потока: открыть список ЛР

Пользователь открывает:

```text
/lab-head/lab-works/?q=7
```

Поток:

```text
GET /lab-head/lab-works/?q=7
  -> LabHeadLabWorksView
  -> LabHeadRequiredMixin.dispatch()
  -> проверка роли LAB_HEAD
  -> проверка учебного центра
  -> get_queryset()
  -> staff_managed_lab_works_qs(user)
  -> filter_lab_head_lab_works(qs, "7")
  -> get_context_data()
  -> lab_works.html
```

Что происходит внутри:

1. `LabHeadRequiredMixin` проверяет, что пользователь имеет право быть в кабинете завлаба.
2. `get_queryset()` получает ЛР, доступные пользователю.
3. `filter_lab_head_lab_works()` применяет поиск.
4. `get_context_data()` добавляет в шаблон дисциплины, лаборатории, аудитории и текущий поисковый запрос.
5. `lab_works.html` рисует таблицу и формы.

## 6. Пример потока: создать ЛР

Пользователь нажимает `Добавить ЛР`.

Открывается модальное окно.

Форма отправляет POST-запрос:

```text
POST /lab-head/lab-works/create/
```

Поток:

```text
POST /lab-head/lab-works/create/
  -> LabHeadLabWorkCreateView.post()
  -> request.POST
  -> проверка discipline
  -> проверка laboratory
  -> проверка default_room
  -> проверка number/duration/capacity
  -> проверка дубля ЛР
  -> LabWork.objects.create(...)
  -> lab_work.laboratories.set(...)
  -> sync_training_centers_for_laboratories(lab_work)
  -> redirect("lab-head-lab-works")
```

Простыми словами:

1. View читает данные формы.
2. Проверяет, что дисциплина доступна завлабу.
3. Проверяет, что лаборатория доступна завлабу.
4. Проверяет, что аудитория относится к нужному УЦ.
5. Проверяет числа.
6. Создает объект `LabWork`.
7. Привязывает ЛР к лаборатории.
8. Обновляет список учебных центров у ЛР.
9. Возвращает пользователя обратно на список ЛР.

## 7. Пример потока: редактировать ЛР

На странице `lab_works.html` каждая строка таблицы хранит данные ЛР:

```html
<tr
    class="lab-work-row"
    data-lab-work-id="{{ lw.pk }}"
    data-title="{{ lw.title|escapejs }}"
    data-number="{{ lw.number }}"
    data-discipline-id="{{ lw.discipline_id }}"
>
```

Когда пользователь кликает по строке:

```text
click по строке
  -> JavaScript openEditDialog(row)
  -> берет data-* из строки
  -> заполняет форму
  -> подставляет action формы
  -> открывает dialog
```

После сохранения:

```text
POST /lab-head/lab-works/<pk>/update/
  -> LabHeadLabWorkUpdateView.post()
  -> lab_head_lab_work_in_scope(user, pk)
  -> проверка полей формы
  -> lab_head_update_lab_work(...)
  -> redirect("lab-head-lab-works")
```

Самое важное: обновление делается через `lab_head_update_lab_work()`, потому что там лежат бизнес-правила.

## 8. Что делает service-слой

Service-слой нужен, чтобы не держать все правила прямо во view.

Если объяснить очень просто:

- view спрашивает: "что пользователь хочет сделать?";
- service отвечает: "можно ли это сделать и с какими объектами?".

### `is_lab_head_user`

Проверяет роль пользователя:

```python
def is_lab_head_user(user: User) -> bool:
    return user.is_authenticated and user.role == UserRole.LAB_HEAD
```

### `lab_head_training_center`

Возвращает учебный центр завлаба.

Если у пользователя в профиле есть лаборатория, учебный центр берется из лаборатории.

Если лаборатории нет, берется `profile.training_center`.

### `lab_head_laboratory`

Возвращает лабораторию завлаба.

Если в профиле явно указана лаборатория, используется она.

Если лаборатории нет, но есть учебный центр, берется первая лаборатория этого учебного центра.

Это удобно, но может быть рискованно, если в одном УЦ несколько лабораторий.

### `sync_training_centers_for_laboratories`

Синхронизирует учебные центры объекта по его лабораториям.

Пример:

```text
ЛР привязана к лаборатории "Электроника"
Лаборатория "Электроника" находится в УЦ №21
Значит у ЛР в training_centers должен быть УЦ №21
```

Функция делает именно это:

```python
tc_ids = obj.laboratories.values_list("training_center_id", flat=True).distinct()
obj.training_centers.set(tc_ids)
```

### `lab_head_people_qs`

Возвращает людей, которых видит завлаб.

Фильтр:

- только роли `LAB_ADMIN` и `TEACHER`;
- если есть лаборатория, фильтрует по `profile__laboratory`;
- если лаборатории нет, но есть УЦ, фильтрует по `profile__training_center`.

### `lab_head_lab_work_search_q`

Строит поиск по ЛР.

Ищет по:

- названию;
- описанию;
- названию дисциплины;
- названию лаборатории;
- названию учебного центра;
- номеру аудитории;
- статусу публикации;
- номеру ЛР;
- длительности;
- вместимости.

Примеры:

```text
q=сопротив
  -> ищет ЛР с "сопротив" в названии, описании, дисциплине и т.д.

q=7
  -> ищет текст "7"
  -> плюс ищет number=7, duration_minutes=7, capacity=7

q=да
  -> ищет опубликованные ЛР

q=нет
  -> ищет неопубликованные ЛР
```

### `filter_lab_head_lab_works`

Применяет поиск к queryset ЛР.

Если поисковая строка пустая, возвращает исходный queryset.

Если строка есть, делает `.filter(...).distinct()`.

`distinct()` нужен, потому что при поиске по many-to-many связям одна и та же ЛР может появиться несколько раз.

### `lab_head_stand_search_q`

Строит поиск по стендам.

Ищет по:

- названию;
- инвентарному номеру;
- описанию;
- аудитории;
- номеру учебного центра, если запрос состоит из цифр.

### `lab_head_bindable_disciplines_qs`

Возвращает дисциплины, которые можно привязать к лаборатории.

Логика:

```text
взять дисциплины активного семестра
исключить те, которые уже привязаны к лаборатории завлаба
отсортировать по названию
```

### `lab_head_bindable_lab_works_qs`

Возвращает ЛР, которые можно привязать к лаборатории.

Логика:

```text
взять дисциплины, которыми управляет завлаб
найти ЛР этих дисциплин
исключить ЛР, уже привязанные к лаборатории
```

### `lab_head_*_in_scope`

Это функции объектной проверки доступа.

Примеры:

- `lab_head_room_in_scope(user, room_id)`
- `lab_head_laboratory_in_scope(user, laboratory_id)`
- `lab_head_discipline_in_scope(user, discipline_id)`
- `lab_head_lab_work_in_scope(user, lab_work_id)`

Они отвечают на вопрос:

```text
Этот объект принадлежит зоне доступа завлаба?
```

Если да - возвращают объект.

Если нет - возвращают `None`.

### `lab_head_update_lab_work`

Это главная функция обновления ЛР.

Она проверяет:

- название не пустое;
- номер ЛР не меньше 1;
- длительность не меньше 30 минут;
- количество мест не меньше 1;
- дисциплина доступна завлабу;
- лаборатория доступна завлабу;
- аудитория относится к учебному центру лаборатории;
- нет другой ЛР с таким же номером в этой дисциплине.

Потом она:

1. записывает новые значения в объект;
2. сохраняет объект;
3. привязывает ЛР к лаборатории;
4. синхронизирует учебные центры;
5. если изменилась вместимость, обновляет вместимость открытых слотов.

## 9. Проверки прав

### Главная проверка страницы

Главная проверка находится в `LabHeadRequiredMixin`.

Он наследуется от `LoginRequiredMixin`.

Это значит:

- сначала Django проверяет, что пользователь залогинен;
- потом код проверяет, что пользователь именно завлаб.

Логика:

```python
if not is_lab_head_user(request.user):
    messages.error(request, "Доступ только для заведующего лабораторией.")
    return redirect("home")
```

То есть если пользователь не `LAB_HEAD`, он не попадет на страницу.

Потом проверяется учебный центр:

```python
if not lab_head_training_center(request.user):
    messages.error(request, "В профиле не указана лаборатория.")
    return redirect("home")
```

### Объектные проверки

Главной проверки роли недостаточно.

Например, завлаб может быть настоящим завлабом, но попытаться отправить POST на чужую ЛР.

Поэтому есть проверки вида:

```python
lab_work = lab_head_lab_work_in_scope(request.user, pk)
if not lab_work:
    messages.error(request, "Лабораторная работа недоступна.")
    return redirect("lab-head-lab-works")
```

Смысл:

```text
Найди ЛР с таким pk,
но только среди ЛР, доступных этому пользователю.
Если такой ЛР нет - не давай ее менять.
```

Такие проверки есть для:

- ЛР;
- дисциплин;
- лабораторий;
- аудиторий;
- учебных центров;
- преподавателей.

## 10. Шаблоны подробнее

### `home.html`

Очень простая главная страница.

Она получает из view счетчики:

- `people_count`;
- `disciplines_count`;
- `lab_works_count`;
- `stands_count`;
- `schedule_count`.

Потом рисует карточки-ссылки.

### `bindings.html`

На этой странице две основные зоны:

1. Список дисциплин лаборатории.
2. Модальное окно привязки дисциплины.

Каждая дисциплина отображается как `details`.

Это HTML-элемент, который можно раскрывать и закрывать.

Внутри дисциплины показываются:

- название;
- семестр;
- количество ЛР;
- список уже привязанных ЛР;
- список ЛР, доступных для привязки.

JavaScript делает три вещи:

- открывает/закрывает модальное окно;
- фильтрует дисциплины внутри модального окна;
- спрашивает подтверждение перед отвязкой дисциплины.

### `lab_works.html`

Это самая развитая страница.

Основные блоки:

1. Header с кнопкой `Добавить ЛР`.
2. Форма поиска.
3. Таблица ЛР.
4. Dialog создания ЛР.
5. Dialog редактирования ЛР.
6. JavaScript.

В таблице каждая строка кликабельная:

```html
<tr class="lab-work-row" tabindex="0" role="button">
```

`tabindex="0"` позволяет выбрать строку клавиатурой.

`role="button"` говорит вспомогательным технологиям, что строка ведет себя как кнопка.

JavaScript поддерживает:

- клик мышью;
- Enter;
- Space.

### `stands.html`

Страница похожа на `lab_works.html`, но проще.

Есть:

- поиск;
- таблица стендов;
- dialog создания стенда;
- JavaScript для открытия/закрытия dialog.

- поиск работает по названию;
- поиск работает по дисциплине;
- поиск работает по номеру;
- поиск работает по аудитории;
- если ничего не найдено, есть текст `ничего не найдено`.

### Стенды

Проверяется:

- можно создать стенд;
- поиск работает по названию;
- поиск работает по инвентарному номеру;
- поиск работает по аудитории;
- если ничего не найдено, есть текст `ничего не найдено`.

### Расписание

Проверяется:

- можно создать `ScheduleEntry` через POST-запрос.

Важно: UI-шаблон расписания сейчас заглушен, но backend-обработчик и тест уже есть.

### Ограничение обычного staff

Есть тест, что обычный сотрудник не может создать стенд через staff-url.

Смысл:

```text

  -> количество стендов не меняется
```

## 12. Возможные баги и риски

### Риск 1. Новый сотрудник создается без лаборатории

В `LabHeadPersonCreateView` новому пользователю выставляется `training_center`, но не выставляется `laboratory`.

Упрощенно:

```python
user.profile.training_center = tc
user.profile.save(update_fields=["training_center"])
```

Но `lab_head_people_qs()` при наличии лаборатории фильтрует людей по `profile__laboratory`.

Из-за этого возможна ситуация:

```text
Завлаб создал сотрудника
  -> сотрудник получил training_center
  -> но не получил laboratory
  -> потом список сотрудников фильтруется по laboratory
  -> новый сотрудник может не попасть в список
```

### Риск 2. Проверяется УЦ, но сообщение говорит про лабораторию

В `LabHeadRequiredMixin` проверяется `lab_head_training_center(request.user)`.

Но сообщение говорит:

```text
В профиле не указана лаборатория.
```

Это может путать.

Фактически код проверяет наличие учебного центра, а не строго лаборатории.

### Риск 3. Если лаборатория не указана, берется первая лаборатория УЦ

`resolve_staff_laboratory()` работает так:

```text
если profile.laboratory есть
  -> вернуть ее
иначе если есть training_center
  -> вернуть первую лабораторию этого УЦ
```

Это может быть опасно, если в одном УЦ несколько лабораторий.

Пользователь без явно указанной лаборатории может получить доступ к первой лаборатории УЦ.

### Риск 4. Некоторые `int(...)` вызываются до `try`

Например, код может делать:

```python
discipline = lab_head_discipline_in_scope(request.user, int(discipline_id))
```

Если вручную отправить `discipline=abc`, то `int("abc")` вызовет ошибку.

Обычная HTML-форма так не сделает, но прямой POST-запрос может.

Лучше такие преобразования держать внутри `try/except`.

### Риск 5. Create и update проверяют ЛР по-разному

В update используется `lab_head_update_lab_work()`, где есть строгие проверки:

- номер >= 1;
- длительность >= 30;
- вместимость >= 1.

В create часть проверок находится прямо во view и выглядит слабее.

Риск:

```text
создание и редактирование одной сущности могут вести себя по-разному
```

Лучше вынести создание ЛР в service-слой по аналогии с обновлением.

### Риск 6. Backend есть, но UI заглушен

Для `people` и `schedule` backend-логика частично есть.

Но шаблоны показывают "На доработке".

Это значит:

- тесты могут проходить через POST;
- backend может создавать данные;
- но обычный пользователь не видит полноценный интерфейс.

### Риск 7. Слишком широкая привязка дисциплин

`lab_head_bindable_disciplines_qs()` берет все дисциплины активного семестра, которые еще не привязаны к лаборатории.

Если по бизнес-логике завлаб должен видеть не весь каталог, а только часть дисциплин, это слишком широкий доступ.

### Риск 8. Тесты не покрывают плохие POST-значения

Сейчас тесты проверяют нормальные сценарии.

Но полезно добавить проверки:

- чужой `discipline_id`;
- чужой `lab_work_id`;
- чужой `room_id`;
- нечисловой `discipline`;
- нечисловой `capacity`;
- завлаб без лаборатории;
- завлаб без УЦ;
- УЦ с несколькими лабораториями.

## 13. Короткое резюме

Архитектура выглядит так:

```text
urls_web.py
  -> views/lab_head.py
  -> services/lab_head.py
  -> models/querysets
  -> templates/bookings/lab_head/*.html
```

Что уже хорошо:

- есть отдельный service-слой;
- есть проверка роли `LAB_HEAD`;
- есть queryset-ограничения по лаборатории/УЦ;
- есть объектные проверки доступа;
- есть тесты на основные сценарии;
- UI для ЛР и стендов уже рабочий;
- поиск по ЛР и стендам покрыт тестами.

Что стоит улучшить:

- выставлять лабораторию при создании сотрудника;
- усилить валидацию POST-данных;
- сделать создание и обновление ЛР одинаковыми по правилам;
- дописать UI для сотрудников и расписания;
- добавить тесты на чужие ID и плохие POST-значения;
- явно решить, можно ли завлабу без лаборатории автоматически брать первую лабораторию УЦ.
